# Design Decisions & Notes

This document details the reasoning behind architectural, design, and product decisions made during the development of the ChargeSquare EV Charging backend case study.

## Open Design Decisions (Stage 1)

*   **Language & Framework**: I chose **Java 21** with **Spring Boot 3.5** because I know how to use them and it is the stack that is recommended. In general it is a good stack for building backend apps
*   **Tariff on a Session (Snapshot vs Look-up)**: **Snapshot at start**. A snapshot is critical because if pricing rules change mid-session, the user must be billed according to the exact price agreed upon at the moment they started charging, ensuring consistent and fair billing.
*   **Service Comms (Sync REST vs One Event)**: **Synchronous REST**. An event-driven architecture (e.g., using RabbitMQ/Kafka) was deemed unnecessary for this minimal scope. Since I did not make wallet service a seperate service I could not also justify an event driven architecture. In a real production environment where we will have many services like notification wallet etc. such an action can be justified.
*   **Dependency-Down Behaviour**: **Fail Fast**. If the Session Service cannot reach the Station Service or Wallet dependencies, it fails fast and returns a clean, correct error response (e.g., `500` or `503`). This avoids the complexity of fallback logic while keeping the system's state predictable. However in session start endpoint I went with a different approach. At first session start was also going to do the same. I was going to first occupy the station then create session and we will be fine in case occupying fails since session start will fail fast. But this would result in inconsistent behaviour in case session creation fails. Therefore instead I first checked whether connector is already occupied or not then started the session then I actually tried to occupy the endpoint. What I wanted to do was deleting the session if occupying fails but I decided to add a FAILED session state for debugging reasons since such an instance will mean either our occupy endpoint fails for some wrong config or something happened between 2 calls. I do realize this introduces a contrast but I did it bc I was required to not create a session if connector is already occupied and I wanted failed sessions.
*   **Repo Layout**: **Monorepo**. Keeping both services (Session and Station) and the shared infrastructure in a single repository is the simplest and most effective approach for a take-home study, facilitating easy testing, shared CI pipelines, and a unified Docker Compose setup.
*   **Database**: **PostgreSQL** because it is a mature, reliable relational database that integrates well with Spring Boot and supports the features required by the project. Also it was the recommended one for the case study. Although this is a microservice-based application, I used a **shared PostgreSQL instance with separate schemas** for each service (`station` and `session`). This keeps deployment simple while still maintaining clear ownership of data. Each service only accesses its own schema through its own repositories, making it straightforward to split them into separate databases in the future if needed. The database schema is created automatically from scratch using **Flyway migrations** and persists across application restarts through a Docker volume.
* **Wallet Placement**: **keep the wallet inside the Session Service** instead of creating a separate Wallet Service. Since the wallet is only used during session start/stop and top-up operations, introducing another microservice would have added extra network communication and complexity without providing significant benefits for this assignment. In a larger production system, a dedicated Wallet Service could provide better separation of concerns, independent scaling, and a complete transaction history, but for the current scope keeping it within Session Service results in a simpler and easier-to-understand architecture. For this reason I also did not go for an event based structure


## System Resilience & Edge Cases

### Idempotent Retries
If the client app resends a "Stop" request because the first response was lost over the network, the current system safely rejects the duplicate request with a `409 Conflict` (because the session is no longer `ACTIVE`), avoiding a double-charge. However, to make this truly safe and idempotent for the caller, we would have to introduce an **idempotency key** (e.g., a unique UUID generated by the client having a timestamp in it (most uuids already have it tho)). The server would store this key alongside the completed session; if it detects a duplicate key, it would simply return the cached `200 OK` receipt of the previously processed stop action instead of throwing a state error.

### Stuck Connectors / Partial Failure
If the Station Service successfully marks a connector as OCCUPIED but the Session Service fails before creating the corresponding charging session, the connector may remain occupied even though no active session exists. In this case study I mark the session as failed and document the limitation. In a production system I would recover from this using short request timeouts, a limited number of retries for transient failures, and a scheduled cleanup job that periodically detects occupied connectors without matching active sessions and returns them to the AVAILABLE state. The trade-off is that aggressive cleanup may accidentally release a connector that is still legitimately in use, while conservative cleanup increases recovery time.

### Distributed Systems Machinery
I intentionally kept the communication between services synchronous and simple. If a required dependency is unavailable during a start or stop operation, the request fails with a clear error rather than attempting automatic recovery. I did not implement retries, exponential backoff, message brokers, sagas, exactly-once delivery, service meshes, caching, rate limiting, or refresh tokens because they would add considerable complexity without providing significant value for a project of this size. These mechanisms become worthwhile as the system grows, traffic increases, or stronger availability and consistency guarantees are required.

As the system evolves, some of these mechanisms would become natural additions. Timeouts, retries with exponential backoff, and circuit breakers would improve resilience against temporary network or service failures. Event-driven communication combined with saga or compensation patterns could help coordinate operations that span multiple services and make services more loosely coupled while ensuring eventual consistency after partial failures. On top of that addition and removal of services becomes easier which will make development significantly more efficient (since they are loosely coupled now). Rate limiting and caching would improve scalability under heavy load, while refresh tokens and a dedicated identity provider would strengthen authentication and user experience in larger deployments. Having a refresh token also allows us to give a shorter expiration time for the current access token which will help with reducing the impact of an allocated access token while maintaining a smooth user experience. These additions were intentionally omitted to keep the implementation focused on the requirements of the assignment while maintaining a clean and understandable architecture.

## Architectural & Domain Notes

*   **Session Stop Ordering**: During completion, the system first attempts to release the connector. Only after the connector is successfully released does the service debit the wallet and mark the session as completed. This strict ordering satisfies the business requirement that a failed connector release must *not* charge the user or prematurely terminate the session. 
*   **Session Start & `FAILED` State**: When starting a session, the system first checks connector availability, starts the session in the DB, and *then* occupies the connector. If occupying the connector fails (e.g., race condition), the session is marked as `FAILED`. Rather than deleting the session row entirely, retaining the `FAILED` state provides essential traceability for debugging misconfigurations or concurrency issues.
*   **Wallet Implementation**: The wallet is implemented minimally as a `wallet_balance` column on the User entity (`BigDecimal`, precision 12, scale 2) to avoid over-engineering. Top-ups currently have no checks, but in reality, they would integrate with a payment system gateway.
*   **Negative Wallet Balances**: Wallet deduction at the end of a session can be problematic. Currently, customers are allowed to hit a negative balance (debt) unless company policy dictates otherwise. In reality, payments are processed via pre-authorizing a set amount or auto-ending a session when the balance hits zero. Handling the latter in a multi-session concurrent environment introduces high complexity, so we allow negative balances for simplicity in this slice.
*   **Zero kWh Sessions**: Sessions reporting `0 kWh` energy are considered invalid and rejected (`400 Bad Request`), assuming a charger malfunction where no start fee should be collected.
*   **Admin vs. Driver Roles**: Admins are treated as actual administrative personnel, not drivers. Admins can see and stop any session, occupy/release connectors manually, and top up any wallet. However, admins *cannot* start a session on behalf of another user.
*   **Internal Occupy/Release APIs**: The `/connectors/{id}/occupy` and `/release` endpoints should not be accessible to the general userbase. While currently secured under the Admin role, in a real system they would be locked down entirely (e.g., via internal network segregation or a service-to-service API key) so only the Session Service could invoke them.
*   **Session Visibility**: The `/users/{userId}/sessions` endpoint currently allows anyone to view sessions for any user ID. This was left open intentionally to facilitate easier local debugging, but would strictly enforce user-ownership checks in production.
*   **Database Seeding**: The population of the database differs between the services because seeding Users requires BCrypt password hashing logic, unlike standard relational data. Therefore seeding for users is done with a java data init script in session-service\src\main\java\com\yusufakdogan\session_service\config\DataInitializer.java.
*   **Observability**: Key actions (session start/stop, wallet debit/top-up, connector occupy/release) are explicitly logged to provide a traceable sequence of events across containers.

## SECURITY

*   **Authentication Approach**: Authentication is handled via a tiny auth endpoint (`/auth/login`) inside the Session Service that issues stateless JWTs upon verifying BCrypt-hashed credentials. The token carries the `userId`, `username`, and `role`.
*   **Role-Based Access Control (RBAC)**:
    *   `VIEWER`: Read-only access. Can list stations, connectors, and view sessions.
    *   `ADMIN`: Write access. Can start on their own behalf, can stop any existing session, top up wallets, and manually occupy/release connectors.
*   **API Protection**: Endpoints are secured using Spring Security's `@PreAuthorize` annotations. Unauthenticated requests receive `401 Unauthorized`, and under-privileged requests receive `403 Forbidden`.
*   **Token Validation**: The token is validated using a shared symmetric secret (`JWT_SECRET`) across services. This avoids an extra network hop for validation. As the services grow, this would be migrated to asymmetric keys (JWKS) or handled centrally by an API Gateway.
*   **Roles in Token Claims vs. Server-Side**: We store the role directly in the JWT claims. This avoids a database lookup on every request, making it highly scalable, but carries the trade-off that role changes don't take effect until the token expires. Currently expiration time is set to 1 hour (it is settable in .env). Refresh tokens could be implemented to mitigate this but they add significant complexity.
*   **Token Storage & Security**: The SPA client should store the token in memory or a secure `HttpOnly` cookie. Storing in `localStorage` exposes the token to XSS attacks. We never trust client-side validation alone because anyone can bypass the UI to call the API directly; all inputs are strictly re-validated on the server.
*   **Secrets Management**: Secrets (`JWT_SECRET`, DB passwords) are injected via environment variables. In a real Kubernetes deployment, these would be securely managed via `Secret` resources or an external Vault, and are never committed to git.
*   **Audit Logging**: Security-relevant actions (who started/stopped a session, topped up a wallet, and failed logins) should be logged with a timestamp and actor ID. This is crucial for an ops panel to trace administrative actions.
